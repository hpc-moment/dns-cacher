# PROTOCOL_WEEK2

## Week 2. Standard cache experiments

### W2-01. 1.1.А Проверка стандартного TTL-кэширования

Для проверки базового кэширования Unbound были выполнены два последовательных запроса к `yandex.ru` через локальный резолвер. Между запросами была выдержана пауза 20 секунд. В первом и втором ответах резолвер возвращал статус `NOERROR`, а значение TTL во втором ответе оказалось меньше, чем в первом. Это подтверждает, что запись находится в кэше резолвера, а её TTL убывает со временем. 

Таким образом, подпункт `1.1.А` считается выполненным.

### W2-02. 1.1.Б Проверка prefetch

Для проверки механизма prefetch использовался домен `example.com`. Начальное значение счётчика prefetch в статистике Unbound было равно `0`. После первичного запроса был выполнен мониторинг TTL записи, который показал его снижение с `2m10s` до `27s`. Это значение ниже 10% от исходного TTL `5m30s`, что соответствует условию срабатывания prefetch. 

После выполнения триггерного запроса при остаточном TTL `27s` резолвер вернул обновлённый ответ с TTL `5m29s`, а затем follow-up запрос показал TTL `5m21s`. Дополнительно статистика Unbound изменилась с `total.num.prefetch=0` до `total.num.prefetch=1`, что подтверждает срабатывание механизма фонового обновления кэша. 

Таким образом, подпункт `1.1.Б` считается выполненным.

### W2-03. 1.1.В Принудительный min/max TTL

Для проверки механизма принудительного минимального TTL был использован домен `api.telegram.org`, который у апстрим-резолвера `8.8.8.8` возвращал TTL `3m8s`. Перед локальной проверкой кэш Unbound для данного домена был очищен командой `unbound-control flush api.telegram.org`. 

После этого первый запрос к локальному резолверу `@127.0.0.1` вернул TTL `5m30s`, а повторный запрос — `5m29s`. Это подтверждает, что Unbound применяет настройку `cache-min-ttl` и повышает TTL записи выше исходного значения, полученного от апстрима. 

Таким образом, подпункт `1.1.В` считается выполненным.

### W2-04. 1.1.Г Вытеснение записей из кэша при ограниченном размере

Для проверки вытеснения записей из кэша сначала была закэширована запись `yandex.ru`, после чего был запущен скрипт массового заполнения кэша `fill_cache.sh`, генерирующий обращения к уникальным поддоменам. В процессе выполнения скрипта фиксировались статистика Unbound и наличие записи `yandex.ru` в кэше. 

После 500 и 1000 запросов запись `yandex.ru` ещё присутствовала в кэше. После 1500 запросов скрипт зафиксировал состояние `yandex.ru в кэше: 0 записей`, а также вывел сообщение `EVICTION ОБНАРУЖЕН: yandex.ru вытеснен из кэша!`. При этом размер кэша по статистике Unbound увеличился с `mem.cache.message=80002` до `mem.cache.message=539395`. Это подтверждает, что при ограниченном размере кэша запись может быть вытеснена до истечения её TTL. 

Таким образом, подпункт `1.1.Г` считается выполненным.
