# PROTOCOL_WEEK3

## Week 3. External Redis cache

### W3-01. 1.2.А Подключение внешнего Redis-кэша

Для проверки подключения внешнего Redis-кэша был использован DNS-прокси, работающий на порту `5300/udp` и взаимодействующий с контейнером `dns-cache`. После запуска стенда был выполнен запрос `dig @127.0.0.1 -p 5300 example.com`. Запрос завершился корректным ответом со статусом `NOERROR`. После этого в Redis появился ключ `dns:example.com.:A`, а для него был зафиксирован положительный TTL. Это подтверждает, что DNS-ответы успешно сохраняются во внешний Redis-кэш. 

Таким образом, подпункт `1.2.А` считается выполненным.

### W3-02. 1.2.Б Проверка механизма кэширования через Redis

Для проверки механизма кэширования через Redis были выполнены повторные запросы к `example.com` через DNS-прокси. В логах `dns-proxy` была зафиксирована последовательность `MISS` для первого запроса и `HIT` для повторного запроса по тому же ключу `dns:example.com.:A`. Дополнительно было подтверждено, что TTL ключа в Redis остаётся положительным и уменьшается со временем, а клиентский запрос через порт `5300/udp` продолжает возвращать корректный ответ со статусом `NOERROR`. 

Таким образом, подпункт `1.2.Б` считается выполненным.

### W3-03. 1.2.В Реализация ответов из Redis

Для проверки обслуживания запроса напрямую из Redis-кэша перед экспериментом были удалены ключи `dns:example.com.:A` и `dns:yandex.ru.:A`. После этого были выполнены два последовательных запроса к `example.com` через DNS-прокси. Первый запрос завершился ответом `NOERROR` и в логах `dns-proxy` был отмечен как `MISS`. Второй запрос также завершился ответом `NOERROR`, но уже был отмечен в логах как `HIT`. Журнал Unbound показал только одно обращение к `example.com` от контейнера `dns-proxy`, что соответствует первому запросу. Повторного обращения к локальному резолверу для второго запроса не зафиксировано. Это подтверждает, что второй ответ был выдан напрямую из Redis-кэша. 

Таким образом, подпункт `1.2.В` считается выполненным.

### W3-04. 1.2.Г Настройка кастомного времени хранения в Redis

Для проверки кастомного времени хранения был сопоставлен TTL прямого ответа локального резолвера и TTL ответа через DNS-прокси. При прямом запросе к `example.com` возвращался TTL около `5m19s`. При запросе к тому же домену через порт `5300/udp` ответ возвращался с TTL около `59m49s`. Дополнительно через `redis-cli` был подтверждён положительный TTL ключа `dns:example.com.:A`, соответствующий примерно одному часу хранения. Это подтверждает, что DNS-прокси сохраняет запись во внешний Redis с собственным временем жизни, отличным от исходного TTL DNS-ответа. 

Таким образом, подпункт `1.2.Г` считается выполненным.
