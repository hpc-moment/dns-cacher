# ADR-002: Контейнеризированный DNS-резолвер Unbound с использованием Systemd

## Статус
Принято

## Дата
2026-03-22

## Контекст
Развертывание DNS-резолвера Unbound требует надежного управления службами, что обычно выполняется с помощью `systemd`. Однако запуск `systemd` внутри Docker-контейнера представляет собой проблему из-за cgroups, специфичных для оборудования целей (targets) и управления журналами. Нам нужно решение, которое позволит Unbound работать как служба systemd внутри Docker-контейнера, оставаясь при этом совместимым с любым хостом, корректно обрабатывая инициализацию и обеспечивая доступность журналов через стандартные команды Docker.

## Решение
Мы решили контейнеризировать систему DNS-резолвера Unbound с использованием Docker со следующей архитектурой:
1. **Базовый образ Ubuntu 22.04:** Используется в качестве основы с установленным `systemd`.
2. **Интеграция Systemd:** Контейнер запускается с `privileged: true` и монтированием `tmpfs` (`/run`, `/run/lock`, `/tmp`), чтобы позволить `systemd` управлять процессами нативно.
3. **Хак для совместимости:** Мы активно удаляем аппаратные и UI-специфичные цели `systemd` (например, `multi-user.target.wants`, `local-fs.target.wants`, `udev`, `graphical.target`, `getty.target`) во время сборки Docker. Мы также явно задаем цель по умолчанию как `multi-user.target`.
4. **Перенаправление журналов:** Журналы `systemd` и вывод Unbound перенаправляются в консоль Docker (`stdout`) путем изменения `/etc/systemd/journald.conf.d/docker-console.conf`.
5. **Автоматизированная валидация:** Реализация как внутренней проверки `healthcheck` в Docker (через `dig`), так и внешнего скрипта `check.sh` для автоматической проверки состояния системы, работоспособности службы и соответствия конфигурации.

## Обоснование
- **Управление службами:** `systemd` обеспечивает нативное и надежное управление демоном `unbound`, обрабатывая перезапуски, зависимости и корректное завершение работы.
- **Портативность:** Удаление целей (targets), связанных с физическим оборудованием, и принудительное использование `multi-user.target` предотвращает зависание или сбои `systemd` при попытках взаимодействия с несуществующими устройствами или подсистемами UI, обеспечивая истинную кросс-платформенную совместимость хостов Linux (NixOS, WSL2, Ubuntu).
- **Наблюдаемость:** Перенаправляя журнал в консоль и внедряя автоматизированные проверки работоспособности, мы сохраняем мониторинг производственного уровня.

## Последствия
- Контейнер должен запускаться в режиме `privileged`, что расширяет поверхность атаки. Это приемлемый компромисс для изолированной лабораторной среды DNS.
- `Dockerfile` требует ручной очистки служб `systemd` для сохранения независимости от хоста без использования продвинутых функций Docker Compose, таких как `cgroupns: host`.
- Обновления конфигурации плавно управляются через монтирование `bind` только для чтения, а их целостность проверяется системой автоматической проверки.

## Рассмотренные альтернативы
- **Простой Entrypoint:** Запуск Unbound напрямую через Docker `ENTRYPOINT` или `CMD` без systemd. Рассмотрено, но отклонено, так как это пропускает нативный жизненный цикл демонизации и этапы проверки, которые предоставляют юнит-файлы systemd.
- **Alpine Linux с OpenRC:** Рассмотрено, но отклонено для сохранения стандартного инструментария Ubuntu и соотношения 1:1 с типичным развертыванием производственных виртуальных машин с `systemd`, с которым команды уже знакомы.
