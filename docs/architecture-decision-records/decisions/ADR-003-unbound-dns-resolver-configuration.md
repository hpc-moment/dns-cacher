# ADR-003: Конфигурация сети и безопасности DNS-резолвера Unbound

## Статус
Принято

## Дата
2026-03-22

## Контекст
После настройки контейнеризированного DNS-резолвера Unbound (ADR-002) нам нужно было определить точные профили сети, производительности и безопасности для самого демона unbound. Резолвер DNS должен эффективно обслуживать локальные запросы и запросы определенных подсетей, использовать криптографическую валидацию (DNSSEC), ограничивать доступ и управлять внутренним кэшированием, чтобы гарантировать высокую производительность и безопасность.

## Решение
Конфигурация `unbound.conf` включает следующие ключевые параметры:
1. **Контроль доступа:** Резолвер слушает на `0.0.0.0:53`. Глобальные запросы строго отклоняются (`0.0.0.0/0 refuse`), но разрешены запросы с localhost (`127.0.0.0/8 allow`) и из изолированной лабораторной сети Docker (`10.10.0.0/24 allow`).
2. **Ограничения протоколов:** Включены TCP и UDP для IPv4 (`do-ip4: yes`, `do-udp: yes`, `do-tcp: yes`), тогда как IPv6 явно отключен (`do-ip6: no`).
3. **Оптимизация производительности:**
   - Включена многопоточность (`num-threads: 2`).
   - Распределено выделение памяти для кэшей: кэш сообщений (`64m`), кэш RRset (`128m`) и негативный кэш (`16m`).
4. **Усиление безопасности:**
   - Включено скрытие идентификатора и версии (`hide-identity: yes`, `hide-version: yes`).
   - Строго применяется DNSSEC (`harden-dnssec-stripped: yes`), а модуль `validator` обязателен вместе с `iterator`.

## Обоснование
- **Сетевая изоляция:** Явное разрешение стандартных внутренних сетей при агрессивной блокировке публичных маршрутов предотвращает превращение сервера в открытый вектор DNS-амплификации, следуя лучшим практикам безопасности.
- **Размеры кэша и потоки:** Правильное разделение памяти между кэшем сообщений (заголовки DNS) и кэшем RRset (фактические данные записей, размер которого в 2 раза превышает кэш сообщений) обеспечивает высокую частоту попаданий. Использование 2 потоков позволяет эффективно использовать современные выделения многоядерных контейнеров без чрезмерного переключения контекста.
- **Безопасность:** Скрытие версии смягчает последствия автоматизированного сканирования уязвимостей. Принудительное использование DNSSEC предотвращает отравление кэша (cache poisoning) и атаки "человек посередине" (man-in-the-middle), обеспечивая целостность разрешенных запросов.

## Последствия
- Работа резолвера ограничена только IPv4. Любые внутренние лабораторные компоненты, строго полагающиеся на разрешение DNS IPv6, потребуют корректировок.
- Большие размеры памяти кэша (`>200MB` в сумме) означает, что контейнер будет потреблять значительный объем ресурсов. Мониторинг узла-хоста должен учитывать этот базовый уровень.
- `remote-control` через порт 8953 требует внутреннего доступа к localhost для выполнения команд вроде `unbound-control status`.

## Рассмотренные альтернативы
- **Публичный открытый резолвер:** Рассмотрено, но отклонено из-за огромных рисков безопасности, связанных с атаками DNS-амплификации.
- **Без валидации DNSSEC:** Рассмотрено для повышения скорости запросов, но отклонено, так как целостность данных и проверка безопасности перевешивают незначительные улучшения задержки.
