# ADR-005: Разработка кэширующего Python DNS-прокси с использованием Redis

## Статус
Принято

## Дата
2026-04-03

## Контекст
В рамках задач 3 недели (w3_task2-5) возникла необходимость реализовать собственный перехватчик и кэширующий слой поверх существующей инфраструктуры DNS (`dns-lab`). Цель заключалась в программном контроле над кэшем DNS-ответов: обработке промахов (MISS), быстрых ответах из кэша (HIT) и управлении временем жизни (TTL) записей с помощью внешнего хранилища (базы данных), к которому есть удобный административный доступ. 

## Решение
Для решения этой задачи была внедрена связка из самописного DNS-прокси и Redis:
1. **Python DNS Proxy:** Написан легковесный DNS-сервер на Python, прослушивающий запросы (например, на порту 5300 или 53 внутри контейнера) и выступающий фасадом для апстрим-сервера (Unbound).
2. **Redis в качестве in-memory кэша:** Использована база данных Redis для хранения разрешенных DNS-запросов. Кэширование прозрачно использует механизм встроенного ключа TTL в Redis, который соответствует времени жизни DNS-записи. 
3. **Развертывание в Docker Compose:** И Python-прокси, и Redis развернуты в единой сети Docker Compose (`dns-lab`) вместе с остальными компонентами.

## Обоснование
- **Гибкость Python:** Использование Python позволяет легко реализовать логику перехвата, сохранения структуры DNS-пакетов и логирования попаданий (HIT/MISS). 
- **Нативная поддержка TTL в Redis:** Использование команды Redis `EXPIRE` или автоматического TTL при вставке ключа снимает необходимость писать самописный Garbage Collector (сборщик мусора) для устаревших DNS-записей. Записи автоматически удаляются по истечении TTL (например, при достижении -2 в `redis-cli`).
- **Удобство дебага:** С помощью `RedisInsight` появляется возможность проводить визуальный мониторинг и ручное изменение TTL для проверки работы кэша прокси в реальном времени.

## Последствия
- Появляются дополнительные накладные расходы на десериализацию/сериализацию пакетов в Python и сетевые прыжки (network hops) до Redis. Успешный `HIT` обрабатывается довольно быстро (например, за 1-2 мс), но промах `MISS` может оказаться значительно медленнее в сравнении с нативными реализациями (например, первичный запрос занимал до 1681 мс из-за дополнительной обработки и задержек апстрима).
- Требуется корректная конвертация байтовых форматов DNS-сообщений для их надежного хранения в Redis.
